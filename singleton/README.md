# Singleton 패턴

## 개념

클래스 인스턴스를 **오직 하나만** 생성하고, 어디서든 그 인스턴스에 **전역으로 접근**할 수 있게 보장하는 생성(Creational) 패턴. 생성자를 `private`으로 막고, 클래스 스스로 유일한 인스턴스를 관리하며 접근 창구(`getInstance()`)를 제공하는 게 핵심.

## 구조

| 역할 | 설명 | 구현 파일 |
|---|---|---|
| **Singleton** | 유일 인스턴스 생성/보관, 전역 접근점 제공 | `King.kt` |
| **Client** | `getInstance()`로 인스턴스 얻어 사용 | `Main.kt` |

## 구현 방식 비교

| 방식 | 스레드 안전 | 특징 |
|---|---|---|
| `object` 선언 (Kotlin 권장) | O | JVM 클래스 로딩 시점에 초기화, 코드 가장 간결 |
| `lazy(LazyThreadSafetyMode.SYNCHRONIZED)` | O | 지연 초기화 + 락, 기본 `lazy`가 이 모드 |
| Double-checked locking (`@Volatile` + `synchronized`) | O | 지연 초기화 + 락 최소화, Java 스타일 |
| `lateinit var` + null 체크 (`King.kt` 현재 구현) | X | **버그**: non-null 타입에 `== null` 비교 시 초기화 전 접근하면 `UninitializedPropertyAccessException` 발생. null 반환 안 됨 |

Kotlin에서는 대부분 `object King { ... }` 한 줄로 충분 — 언어 차원에서 싱글턴을 지원하기 때문.

## 왜 쓰는가

- **자원 절약**: DB 커넥션 풀, 설정(Config), 로거처럼 무겁거나 상태 공유가 필요한 객체를 하나만 유지
- **전역 상태 일관성**: 여러 곳에서 동일 인스턴스를 봐야 할 때 (게임 내 국왕 하나, 캐시 매니저 등)
- **접근 통제**: 인스턴스 생성 로직을 한 곳에 캡슐화

## 실무 사용

- Spring Bean 기본 스코프(`singleton`)
- 로거(`Logger`), 설정 로더, 캐시 매니저
- DB 커넥션 풀 (`HikariCP` 등)
- Kotlin `object` 키워드 자체가 사실상 싱글턴 패턴의 언어 내장 구현

## 주의점

- **테스트 격리 깨짐**: 전역 상태라 테스트마다 리셋 필요, 실행 순서에 따라 결과 달라질 수 있음. 생성자 주입이 아니라서 목(mock) 대체도 까다로움
- **숨은 의존성 (Hidden Dependency)**: `getInstance()`가 메서드 내부에 숨어있어 클래스 시그니처만 봐선 어떤 협력 객체를 쓰는지 안 보임. 의존 관계 파악하려면 구현 코드까지 뜯어봐야 함 — 생성자 주입(DI) 쓰면 시그니처만으로 파악 가능
- **전역 가변 상태**: 어디서든 접근/변경 가능해 사실상 전역 변수와 다를 바 없음. 어떤 코드가 언제 상태를 바꿨는지 추적하기 어려워 부작용(side effect) 디버깅 힘듦
- **멀티스레드 위험**: 동기화 안 하면 인스턴스 중복 생성 가능. 동기화하면 락 경합(lock contention)으로 병목 생김
- **결합도 증가**: 참조하는 모든 클래스가 구체 클래스에 강결합됨. 인터페이스로 감싸지 않으면 교체·확장·모킹 어려움
- **생명주기 통제 불가**: 앱 종료 전까지 계속 살아있어 GC 안 됨 — 상태 큰 객체면 메모리 누수 원인 될 수 있음
- **직렬화/리플렉션으로 우회 가능**: Java 리플렉션으로 private 생성자 강제 호출하거나 역직렬화 시 새 인스턴스 생김 — `enum` 싱글턴이 이 문제엔 가장 안전
- **DI 컨테이너와 병행 고려**: Spring 등에서 "싱글턴 스코프 Bean"으로 관리하면 위 문제 상당수 완화됨 (테스트 시 다른 스코프로 교체 가능, 의존성도 명시적). 수동 `getInstance()` 구현은 정말 필요할 때만 최후 수단으로
- 남용 시 사실상 전역 변수와 다를 바 없어짐 — 정말 유일해야 하는 객체인지 먼저 검토
