# SOLID 원칙

## 개념

객체지향 설계를 "변경에 강하게" 만들기 위한 5가지 원칙. 각 원칙은 독립적이지만 서로 겹치는 지점이 많음(예: LSP를 지키려 인터페이스를 쪼개면 자연히 ISP도 지켜짐).

이 폴더는 원칙마다 **위반(Bad~) 버전과 준수 버전을 나란히** 둬서 뭐가 왜 문제인지 바로 비교 가능하게 구성. 도메인은 실무에서 흔한 "판매 리포트 / 결제 수수료 / 재고 / 사무기기 / 주문 알림".

## 구조

| 원칙 | 위반 시 문제 | 구현 파일 |
|---|---|---|
| **SRP** (단일 책임) | 계산+포맷+저장+발송이 한 클래스 → 포맷만 바꿔도 전체 재테스트 | `SingleResponsibility.kt` |
| **OCP** (개방-폐쇄) | 결제 수단 추가마다 기존 `when` 분기 수정 → 회귀 위험 | `OpenClosed.kt` |
| **LSP** (리스코프 치환) | 읽기 전용 구현체가 쓰기 메서드에서 예외 던짐 → 다형적 순회 시 런타임 크래시 | `LiskovSubstitution.kt` |
| **ISP** (인터페이스 분리) | 뚱뚱한 인터페이스 강제 구현 → 지원 안 하는 기능에 예외 던지는 죽은 코드 | `InterfaceSegregation.kt` |
| **DIP** (의존성 역전) | 고수준 모듈이 저수준 구현을 직접 생성 → 교체/테스트 어려움 | `DependencyInversion.kt` |

## 왜 쓰는가

- **SRP**: 변경 이유가 하나뿐이면, 그 이유로만 클래스를 고치면 됨 — 다른 책임까지 덩달아 깨질 일 없음
- **OCP**: 새 요구사항을 "추가"로 해결 — 기존에 잘 동작하던 코드를 다시 열어 고치지 않아도 됨
- **LSP**: 부모 타입으로 자식을 다뤄도 안전 — 다형성을 신뢰하고 `instanceof`/타입 체크 없이 코드 작성 가능
- **ISP**: 클라이언트는 자기가 쓰는 메서드만 알면 됨 — 안 쓰는 메서드 때문에 재컴파일/재구현될 일 없음
- **DIP**: 고수준 정책이 저수준 세부사항(SMTP, DB 드라이버 등)에 묶이지 않음 — 테스트 시 가짜 구현 주입 용이

## 실무 사용

- **SRP**: 계산/포맷팅/영속화/알림을 서로 다른 클래스로 분리하고, 서비스 클래스는 조립만 — Spring 등에서 `@Service`가 여러 `@Repository`/`@Component`를 조합하는 방식 자체가 SRP 실전 형태
- **OCP**: 결제 게이트웨이, 파일 포맷 등 "종류가 계속 늘어나는" 도메인은 인터페이스 + 팩토리/맵 조회로 설계해 신규 구현체 추가만으로 대응
- **LSP**: 리포지토리/캐시처럼 읽기·쓰기 성격이 다른 구현체가 섞이는 계층에서 인터페이스를 read/write로 쪼개는 게 흔한 해법
- **ISP**: REST 클라이언트, 외부 SDK 래퍼처럼 기능이 많은 API를 감쌀 때, 클라이언트별로 필요한 기능만 노출하는 좁은 인터페이스를 따로 정의
- **DIP**: 알림/결제/저장소 등 외부 연동은 항상 인터페이스로 추상화하고 구현체는 DI 컨테이너가 주입 — 테스트에서는 목(mock)/스텁으로 교체

## 주의점

- SRP를 과하게 적용하면 클래스가 지나치게 잘게 쪼개져 "책임 하나당 파일 하나"가 오히려 협업을 방해할 수 있음 — 응집도 있는 단위로 묶는 감각 필요
- OCP는 "모든 변경 지점을 미리 확장 가능하게" 설계하라는 뜻이 아님 — 실제로 자주 바뀌는 지점에만 적용(과설계 주의)
- LSP 위반은 컴파일 타임에 안 잡히고 런타임에만 터지는 경우가 많음(`UnsupportedOperationException`) — 코드 리뷰에서 "이 메서드가 부모 계약을 다 지키는가"를 의식적으로 확인해야 함
- ISP로 인터페이스를 너무 잘게 쪼개면 구현 클래스가 여러 인터페이스를 동시에 구현하느라 선언부가 장황해질 수 있음
- DIP는 "인터페이스만 있으면 다 좋다"가 아님 — 구현체가 하나뿐이고 교체 계획도 없는데 인터페이스부터 만드는 건 불필요한 간접 계층
