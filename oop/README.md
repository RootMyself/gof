# 객체지향 프로그래밍 (OOP)

## 개념

데이터(상태)와 그 데이터를 다루는 동작(행위)을 하나의 단위(객체)로 묶고, 객체 간 **협력**으로 프로그램을 구성하는 패러다임. 4대 원칙 — 캡슐화, 추상화, 상속, 다형성 — 은 그 자체가 목적이 아니라 **변경에 강한 코드**를 만들기 위한 도구.

이 폴더 예제는 흔한 "동물이 소리를 낸다" 류 장난감 예제 대신, 실무에서 실제로 마주치는 **주문 처리 시스템**(상품 담기 → 할인 계산 → 결제 → 배송 상태 전이 → 알림 발송)으로 4원칙을 보여줌.

## 구조

| 원칙 | 설명 | 구현 파일 |
|---|---|---|
| **캡슐화** | `Money`는 음수 금액을 생성 시점에 차단, `Order.status`는 public getter / private setter로만 노출 | `Money.kt`, `Order.kt` |
| **추상화** | `DiscountPolicy`는 "무엇을 하는지"(계약)만 노출, "어떻게 계산하는지"는 구현체 내부에 숨김 | `DiscountPolicy.kt` |
| **상속** | `OrderNotifier`가 공통 흐름(`notify`)을 고정하고, `Email/SmsNotifier`가 메시지 형식·발송 채널만 오버라이드 (템플릿 메서드) | `OrderNotifier.kt` |
| **다형성** | `CustomerTier` enum 상수마다 `benefits()` 동작이 다름, `DiscountPolicy` 구현체를 갈아 끼워도 `Order`는 코드 변경 없음 | `CustomerTier.kt`, `DiscountPolicy.kt` |
| **합성** | `Order`는 상속 대신 알림 목록(`List<OrderNotifier>`)과 할인 정책(`DiscountPolicy` 파라미터)을 필드/인자로 조립 | `Order.kt` |
| **상태 캡슐화** | `sealed class`로 가능한 상태를 한정, `when`이 하위 타입을 하나라도 빠뜨리면 컴파일 에러 | `OrderStatus.kt` |

## 왜 쓰는가

- **변경 격리**: 할인 정책이 바뀌어도 `Order`, 알림 로직이 바뀌어도 `Order` — 서로 영향 없음
- **불변식 보장**: 금액이 음수가 되거나, 수량이 0 이하이거나, 배송 완료된 주문 상태가 되돌아가는 일을 타입/생성자/메서드 수준에서 원천 차단
- **의도 노출**: `DiscountPolicy` 인터페이스 하나만 보면 "할인 정책은 subtotal을 받아 할인액을 돌려준다"는 계약이 바로 읽힘
- **테스트 용이**: 각 정책·알림·상태 전이를 독립적으로 단위 테스트 가능

## 실무 사용

이 예제 자체가 실무 축소판:

- **Money 값 객체**: 원시 타입(Double/Long)으로 금액을 다루면 부동소수점 오차, 음수 금액 같은 버그가 생김 — 실제 결제/정산 시스템 대부분 `Money`/`BigDecimal` 래퍼 사용
- **정책 객체 주입**: 할인·배송비·수수료 계산기를 인터페이스로 추상화해 DI 컨테이너(Spring 등)로 주입 — 요구사항 바뀔 때 구현체만 교체
- **상태 머신**: 주문/결제/배송처럼 "되돌릴 수 없는 전이"가 있는 도메인은 sealed class(Kotlin) / enum + 상태 패턴으로 모델링해 잘못된 전이를 컴파일/런타임에 차단
- **템플릿 메서드 상속**: 알림(이메일/SMS/Push), 배치 잡(전/후처리 공통, 본처리만 다름)처럼 "흐름은 같고 일부만 다른" 경우에 한정적으로 상속 사용

## 주의점

- **상속 남용 금지**: 상속 계층이 깊어지면 부모 변경이 모든 자식에 전파되는 취약한 기반 클래스(fragile base class) 문제 발생. 이 예제도 상속은 `OrderNotifier` 한 곳에만 쓰고, 나머지는 인터페이스 + 합성으로 처리
- **할인 중첩 버그**: `CombinedDiscountPolicy`처럼 여러 정책을 단순 합산하면 할인액이 subtotal을 초과할 수 있음 — 실무에서 실제로 자주 나는 버그, 반드시 상한 처리 필요
- **`private set` 생략 주의**: `var status`를 그냥 public으로 두면 아무 코드나 `order.status = OrderStatus.Delivered`로 상태 머신을 깨뜨릴 수 있음
- **God Object 경계**: `Order`에 할인 계산·알림·상태 전이 로직을 전부 넣지 않고 각각 별도 타입(`DiscountPolicy`, `OrderNotifier`, `OrderStatus`)으로 위임했는지 확인 — 그렇지 않으면 클래스 하나가 모든 책임을 떠안는 God Object가 됨
